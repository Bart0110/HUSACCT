What is a Software Architecture Conformance Checking Tool?

When building new software, software architects design an architecture for the application. The programmers are tasked with realizing the application based on the designed architecture.

To see if the realized architecture really is the same as the designed architecture we can use the SACC-tool. To use such a SACC-tool, the following steps are to be taken.

Analyse an architecture

By analyzing an application, we are able to break down the structure of the given software. By doing this the program detects the physical entities inside the software and defines the architecture used within the application.
It also allows the architects and the software engineers to get a graphical representation of the detected software architecture. Next to the possibility of getting a detailed graphical overview of the analyzed software, there is the ability to check the analyzed software architecture with an architectural model.

Define an architecture

Within the tool we define an architecture and the rules that apply to the analyzed application's architecture. We create logical modules such as Layers and define rules that should be followed. The communication rules present in the MVC pattern could be seen as an example to such rules.

Examples:

  1. Logical Layer 'Presentation' is allowed to use Logical Layer 'Task',
    but not with Logical Layer 'Infrastructure'.
  2. Package 'database' is not allowed to use package 'gui'.
  3. Class 'MySQLDatbaseImpl' is not allowed to use class 'UserRegistrationController'.

(not actual syntax/code used in the tool)

Mapping the defined architecture

We then map the physical entities we found, to the defined logical modules. This way the tool knows which physical entities (such as namespaces, packages, classes, interfaces, etc.) are included in logical modules.

Examples:

  1. Package 'gui' should be present in Logical Layer 'Presentation'.
  2. Package 'database.mysql' should be present in Logical Layer 'Infrastructure'.
  3. Class 'UserRegistrationController' should be present in Logical Layer 'Presentation'.

(not actual syntax/code used in the tool)

Executing a conformance check

By executing a conformance check we can see if any of the specified rules are broken. A SACC-Tool automates the process of checking whether or not these rules are broken. The reports from the conformance checks will list/graphically display the violations that occur between entities.

Using the results from this conformance check software-engineers and architects can evaluate their realized software architecture. If any serious issues pops up they can improve the program in such a way, it doesn't violate any rules anymore and into the program the architects meant it to be

Why build a Software Architecture Conformance Checking Tool?

After conducting research into other SACC-tools we found out that the tools aren't always as user friendly and accurate as we had expected them to be. We were then assigned to create our own SACC-tool and improve on some concepts we found lacking in others.

The application we were instructed to build is meant to be a contribution to the field of Software Architecture analysis and evaluation.

Support

The tool itself is written in Java.
Currently there is (only) support for the following languages:

  • Java (Using Antlr)
  • .NET C# (Using Antlr)

We are trying to build support for the following rule types:

  • Unique responsibility
  • Visibility convention
  • Naming convention
  • Subclass convention
  • Interface convention
  • Is allowed to use
    • Is only allowed to use
    • Is the only module allowed to use
    • Must use
  • Is not allowed to
    • use modules in a higher layer
    • use modules in a not directly lower layer
  • Exceptions to all those rules

Also we will try to support the following violation types:

  • Invocation of a Method
  • Invocation of a Constructor
  • Access to a property or field
  • Extending a concrete class
  • Extending a abstract class
  • Implementing a interface
  • Declaration
  • Annotation
  • Import
  • Exception

The architecture should allow for expansions to support more languages and rule types.
